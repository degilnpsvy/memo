2.5.2 Encrypted Configuration File Support

RLX-VM supports encrypted or raw configuration file. RLX-VM always prefer raw configuration file to an encrypted one. When a raw configuration file can’t be opened, RLX-VM will try to read an encrypted one.

Global options open to end user (of RLX-VM) will be found in raw global configuration file, which has the name form as “xxxx_user.lua”, and other global options can be grouped in an encrypted configuration file, which has the name form as “xxxx_chip.lua”. The unencrypted version of configuration file may be kept for development convenience before release to end user. Be sure to define a section (even without any parameters) for all instantiated modules in these two global configure files. Otherwise, RLX-VM may complain it can not find parameters for the instantiated module. Configuration parameters from raw configuration file and encrypted configuration file are accumulated. However, if the single parameter appears in both of the global configure file, the later one takes precedence.

All configuration files of individual model instance are suggested to be encrypted to end user. You may use unencrypted configuration files in development stage and only deliver encrypted configuration files to end user.

The following table shows types of configuration files. Commonly RLX-VM uses 2 global configuration files and many model instance configuration files. Configuration options in rlxvm_config_user.lua are open to end user and options in rlxvm_config_chip.lua are concealed to end user (you should decide which options can be open to end user. You could check the example global configuration files in directory platform/rlxocp for reference.). Options in these 2 global configuration should not overlap and if so, options in rlxvm_config_chip.lua will overwrite these in rlxvm_config_user.lua. Before release to end user, you should generate encrypted configuration files and delete unencrypted ones.

To encrypt or decrypt configuration files, you can use vmenc(for encryption) and vmdec(for decryption) in INSTALL_DIR/bin after RLX-VM is installed. vmenc will not encrypt files which have been encrypted already and vmdec will not decrypt files not encrypted. You may not deliver vmenc and vmdec to end users.

See Chap.1.3.2 for the detailed usage about vmenc and vmdec.
