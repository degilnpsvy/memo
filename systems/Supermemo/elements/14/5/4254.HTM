2.5.4 Generic Engine Configuration Parameters

l SHOW_ACTION

Type: integer

Default: 0

Description: Set this parameter to a positive value to tell generic engine to output the internal action information.

 

l INSTANCES

This is a Lua table, which sets all the model instances of the target system. One item of the table sets one model instance. For example:

{"rlxvm_rlxsim_ocp_tl2_64" , "rlx5181" },

The 1st field, "rlxvm_rlxsim_ocp_tl2_64", of the item is the model type of the instance.

The 2nd field, "rlx5181", is the name of the model instance.

So the example tells the generic engine that there is a model instance, whose name is “rlx5181”, whose type is “rlxvm_rlxsim_ocp_tl2_64”.

 

l EVENTS

This parameter is optional.

This is a Lua table, which sets the event-trigger Configuration of VM. One item of the table sets one event-trigger entry. When certain event is triggered, the relative actions will be done.

Maximum number of event-trigger entry is 2^32. Actions have the same limit.

Some events and actions are generic. Others are specific to a model. Description about the generic events and actions can be found here. Descriptions about specific events and actions can be found in the chapter for the model in this guide.

For detailed information, please check Chapter 7.

 

l CONNECTIONS

 This is a Lua table, which specifies all the connection of the model instances

in the target system. One item of the table specifies one connection. Connection in RLX-VM has a type. The first field of the configuration item sets the connection type. The remaining field of the configuration item may be different for different connection types. There are some predefined connection types:

(I) ocp_tlm2

 This connection is for OCP bus port. For example:

{ "ocp_tlm2", "rlx5181", "bus", "bridge", "bus_s64", "rlx5181"},

The 1st field, “ocp_tlm2”, is the connection type of “ocp_tlm2”.

The 2nd field, "rlx5181", is the name of the instance, who has the master socket.

The 3rd field, "bus", is the name of the master socket.

The 4th field, "bridge", is the name of instance, who has the slave socket.

The 5th field, "bus_s64", is the name of the slave socket.

The 6th field, "rlx5181", is an optional field. If the 6th field is not empty, it means there is an OCP monitor on this OCP connection and the field sets the name of the monitor. The OCP transaction is saved in a file, whose name is the value of the 6th field plus “.ocptrn”.

So the example specifies an OCP connection from the socket “bus” of the instance “rlx5181” to the socket “bus_s64” of the instance “bridge” and there is an OCP monitor “rlx5181”, which will save the OCP transaction in the file “rlx5181.ocptrn”.

After the OCPIP standard TLM monitor, we can attach any number of user-defined OCP TLM monitor. We have rlxvm_ocp_mon_addr_stat (refer 3.18) for now. For example we can attach such a monitor by:

{ "ocp_tlm2", "rlx5181", "bus", "bridge", "bus_s64", "", “rlx5181”},

NOTE: (1) The place for OCPIP standard TLM monitor must be reserved.

(2) Create such a monitor instance before attaching; Add configuration for such a monitor.

Currently, we only support OCP data with 32-bit and 64-bit, number of the binds of the socket is 0 or 1.

 

(II) p_ep_ch

This is the connection which connects a SystemC port and a SystemC export, or connects a SystemC port and a SystemC channel. For example:

{ "p_ep_ch", "ictl", "irq", "rlx5181", "INTREQ_n_2" },

The 1st field, "p_ep_ch", is the connection type of “p_ep_ch”.

The 2nd field, "ictl", is the name of instance, who has the port.

The 3rd field, "irq", is port name.

The 4th field, "rlx5181", is name of the instance, who has the export, or the name of the channel. The example here is the former case.

If the 4th field is the name of the instance, the 5th field, "INTREQ_n_2", is the export name.

So the example means that the port “irq” of the instance “ictl” is connected to the export “INTREQ_n_2” of the instance “rlx5181”.

Currently, we only support SystemC port, which has the interface of

l rlxvm_pin_if_sc

l rlxvm_reset_if_sc

l rlxvm_pcie_tlp_to_core_if

l rlxvm_pcie_core_to_tlp_if

 

(III) p_ch_p

This is the connection to connection two SystemC ports through a SystemC channel. For example:

{ "p_ch_p", "uart0", "tx", "uart2tcp", "rx", "uart_tx"},

The 1st field, "p_ch_p", is the connection type of “p_ch_p”.

The 2nd field, "uart0", is the the name of the instance, who as the source port.

The 3rd field, "tx", is the source port name.

The 4th field, "uart2tcp", is the name of the instance, who has the destination port.

The 5th field, "rx", is the destination port name.

The 6th field, "uart_tx", is the channel name.

So the example means the port “tx” of the instance “uart0” is connected to the port “rx” of the instance “uart2tcp” through the channel “uart_tx”.

Currently, we support channel of:

l sc_signal<bool>, sc_signal<uint8_t>  

l sc_buffer<bool>, sc_buffer<uint8_t>

l sc_fifo<uint8_t>, sc_fifo<ether_packet_t>

 

(IV) osci_tlm

This connection is for I2C bus port which use OSCI TLM simple socket to transfer information. For example:

{ "osci_tlm", "ps_i2c0" , "master" , "ps_i2c_bus" , "bus_m_0" },

The 1st field, " osci_tlm ", is the connection type of “osci_tlm”.

The 2nd field, "ps_i2c0", is the name of the instance, who has the master socket.

The 3rd field, "master", is the name of master socket.

The 4th field, " ps_i2c_bus ", is the name of the instance, who has the slave socket.

The 5th field, "bus_m_0", is the slave socket name.

So the example means the master socket “master” of the instance “ps_i2c0” is connected to the slave socket “bus_m_0” of the instance “ps_i2c_bus”.

User must choose the correct connection type. The port, export, socket of a model is documented in the section for that model.
