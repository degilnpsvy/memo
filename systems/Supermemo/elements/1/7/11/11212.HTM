7.5 # SC_FORK/SC_JOIN

Another use of dynamic threads is dynamic test configuration. This feature is exemplified with a verification strategy sometimes used by Verilog suites using fork/join. Although this technique does not let you create new modules or channels dynamically (because processes may choose to stimulate ports differently on the fly), you can reconfigure tests. Let’s see how this might be done. Consider the DUT in the following figure:

Fig. 7.9  High-level testbench

For each interface (AXI, USB2, etc.), an independent process can be created either to send or receive information likely to be generated in a real system.

Using these processes and fork/join, a high-level test might look like Fig. 7.10 (Note: Syntax will be explained shortly.):

 

 1DataStream d1, d2;

 2SC_FORK

 3sc_spawn(

 4  sc_bind(&dut::AXI_xmt, this, sc_ref(d1)), "p1")

 5, sc_spawn(

 6  sc_bind(&dut::PCIX_rcv, this, sc_ref(d1)), "p2")

 7, sc_spawn(

 8  sc_bind(&dut::USB2, this, sc_ref(d1)),  "p3")

 9, sc_spawn(

10  sc_bind(&dut::HT1_xtm, this, sc_ref(d2)), "p4")

11, sc_spawn(

12  sc_bind(&dut::HT2_rcv, this, sc_ref(d2)), "p5")

13SC_JOIN

Fig. 7.10  Example of fork/join application

Let’s look at an example that involves a number of syntax elements discussed thus far. First, let’s inspect the header for this module in Fig. 7.12.

 

 1//FILE: Fork.h                                                                         

 2SC_MODULE(Fork)

 3{

 4  ...

 5  sc_fifo<double> wheel_lf, wheel_rt;

 6  SC_CTOR(Fork);// Constructor                                                        

 7  // Declare processes to be used with fork/join                                      

 8  void fork_thread();

 9  bool road_thread(sc_fifo<double>& which);

10};

Fig. 7.12  Example header for fork/join example

 

Notice that we pass a FIFO channel by reference so that road_thread can possibly access the FIFO channel. Passing ports or signals by reference would be a natural extension of this idea.

Now, let’s inspect the rest of the code in Fig. 7.13.

 

 1//FILE: Fork.cpp                                                                      

 2#define SC_INCLUDE_DYNAMIC_PROCESSES

 3#include <systemc>

 4#include "Fork.h"

 5...

 6Fork::Fork(sc_module_name nm) //{{{                                                   

 7  : sc_module(nm)

 8{

 9  SC_THREAD(fork_thread);

10  ...

11}

12void Fork::fork_thread()   //{{{                                                      

13{

14  bool lf_up, rt_up; // use for return values                                         

15  SC_FORK

16  sc_spawn(

17    &lf_up

18    , sc_bind(

19      &Fork::road_thread

20      , this

21      , sc_ref(wheel_lf)

22    )

23    , "lf" // process name                                                            

24  )

25  , sc_spawn(

26    &rt_up

28      &Fork::road_thread

30      , sc_ref(wheel_rt)

31    )

32    , "rt" // process name                                                            

33  )

34  SC_JOIN

35}

Fig. 7.13  Example of fork/join

 

The full example may be found in the downloaded examples as Fork. This downloaded example also illustrates the use of sc_spawn instead of SC_THREAD. Using a capitalized word fork avoids collision with the Unix system call fork, which has nothing to do with SystemC’s SC_FORK. Recall that SystemC is a cooperative multitasking system. You should not confuse Unix’s fork facilities with these concepts.
