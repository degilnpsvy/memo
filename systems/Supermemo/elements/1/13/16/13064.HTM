7.3. Formatting

  • Must be consistent: Lack of consistency in code formatting shows a lack of interest in code quality, which probably indicates that a person is not cut out for programming.
  • Each level of indentation should be the same, and done in the same fashion.
  • Use consistent spacing between subprograms, and between smaller blocks within subprograms.
  • Use a neat and consistent format for comments.
  • Matching structural components (e.g. curly braces) must be aligned.
  • High-level languages: Increase indentation inside if/else blocks, loops, etc
  • Assembly language: Use consistent indentation for all opcodes/directives in the code.
  • Assembly language: All labels should begin in the first column. This is a common convention and is required by some assemblers.
  • Keep indentation size reasonable (4 spaces is a good convention for HLLs, 8 for assembly opcodes/directives) so that there's room to the right of statements for brief comments.
  • Whitespace between statements is good for readability, as long as it's within reason, and consistent. Separate subprograms with 2 blank lines, and blocks within subprograms by one blank line. Using more than this means less code can be seen in the editing window.
  • Limit lines to 80 columns. On modern high-res monitors, it is tempting to widen the editing window so you can see more, but this will backfire when you print the code or view it on a smaller display. The number of lines in your editor window is less critical, but well-written code can generally be easily read in an 80x24 terminal window. The better your code's modularity, the less of it you need to see at once. Use this as a litmus test of your code quality.
