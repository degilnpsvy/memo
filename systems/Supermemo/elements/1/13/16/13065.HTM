7.4. Readability

7.4.1. Variables

Identifier (variable, function, label) names must tell the reader exactly what the variable holds or what the code does. Use full words, not abbreviations. Shortcuts tend to backfire as the program grows and similar variables are defined. Names should not contain useless components (e.g. _val, _int, _string).

Example 7.7. Prefixing Variables

Assembly language: Prefix (decorate) all variable names and labels with the name of the procedure they belong to.

	    .data
    # Local variables
    sort_low_index:     .word   0
    sort_start_index:   .word   0
    
	    .text
    sort:
	    sw      $zero, sort_start_index
	    
    sort_next_start:
	    lw      $t0, sort_start_index
    
	    .data
    main_list:      .word   0:100
    main_list_size: .word   0
    
	    .text
    main:
    
	    jal     read_list
	    sw      $v0, main_list_size
	    ...

7.4.2. Constants

All constants must be given descriptive names. The only place in a program where unnamed constants may appear is in constant or variable definitions.

Example 7.8. Named Constants for Readable Code

A cryptic example:

	    li      $t0, 1
    next_fact:
	    blt     $t0, 2, factorial_done
	    mul     $t0, $t0, $a0
	    addi    $a0, $a0, -1
	    j       next_fact

A much more readable example:

    FACT_BASE = 1       # Base case for factorial
    FACT_LOWEST_N       # Lowest N worth multiplying by
    FACT_STEP = -1      # Stride between values to multiply by
    
	    li      $t0, FACT_BASE
    next_fact:
	    blt     $t0, FACT_LOWEST_N, factorial_done
	    mul     $t0, $t0, $a0
	    addi    $a0, $a0, FACT_STEP
	    j       next_fact
