15.2. Introduction

The term exception refers to any hardware event that causes the interruption of normal program flow, without the consent of the running program.

Exceptions are detected by the hardware, but dealt with by software. Handling exceptions can be complex, and the appropriate method of handling them may vary among systems using a particular CPU, so it doesn't make sense to have the hardware do more than detection.

When an exception occurs, the currently running program is interrupted, and a branch automatically occurs to a subprogram called an exception handler. Exception handlers are also known as trap handlers interrupt handlers, or interrupt service routines (ISRs). After the exception handler finished executing, the program that was running when the exception occurred may continue where it left off, completely unaware than an exception occurred. However in cases where the exception was caused by a fatal error, the running program may be terminated.

Britton's language is somewhat inconsistent, so we will be careful to following the definitions below. Exceptions are divided into two subcategories:

  • Traps are exceptions caused by internal events when the CPU itself detects a problem or other reason to interrupt the currently running program. These include, but are not limited to breakpoints, arithmetic overflow and underflow, illegal instructions, bus errors, and segmentation faults. Traps often result in termination of the running program, since bugs in the program are often the cause of the trap.
  • Interrupts are external events, caused by devices outside the CPU. Most interrupts are triggered by input/output devices that either need to send input, or are ready to receive more output. After an interrupt is serviced, the running program resumes where it left off, unaware that anything has happened.
