Address space protection

It is assumed that all memories and other memory mapped peripherals that can be marked as secure are on chip, and therefore encryption is not required in order to protect them. Any accesses to the address space include a new attribute NS-Attr, which is used to mark transactions as non-secure or secure. The conveyance of NS-Attr is IMPLEMENTATION DEFINED.

 

The security extensions require that:

l Areas of the address space marked as secure must only be accessible by transactions with NS-Attr set to secure. Any accesses to secure addresses with NS-Attr set to non-secure must result in an error being reported to the component that generated the request.

l The processor be able to determine the security status of an address (see Security attribution)

 

The security extensions also define a new NS-Req attribute, which defines the security state that the core requests the data transaction be performed in, unless otherwise specified NS-Req is equal to the security state of the core.

 

TrustZone Integration

In some complex SOC’s it may be useful to integrate an M-profile processor with an A-profile core (e.g. the M-profile processor could be used for low level power management). In such systems one possible integration strategy would be to connect NS-Attr from the Cortex-M processor to the existing TrustZone NS-Attr, such that instructions executing in the secure state can access secure locations. This approach is also applicable in systems that re-use TrustZone system IP but do not actually contain a processor with the A-profile Security extensions.

 

Because M-profile processors do not contain an MMU they are not capable of performing any address translation, and are limited a single 4GByte address space. This address space is further restricted by the system memory map as defined in the ARMv7-M specification. In systems where an M-Profile processor coexists with an A-profile processor care must be taken to ensure that the M-profile core can address all memory and peripherals (including both secure and non-secure) that it requires access to. This may require some system level address translation to be performed.

 

MPU extensions

The MPU registers (i.e. MPU_*) are banked with security; For data accesses the MPU memory attribution and privilege checking uses the configuration registers corresponding to the processors currently executing security state. For instruction fetches the MPU uses the configuration registers associated with the security of the address being fetched (see Security attribution). This behavior means that the secure MPU will be used when performing access checks on guard instructions when entering the secure state, which is useful when virtualizing multiple secure unprivileged states.

NOTE: As the banking of the MPU registers includes the MPU_TYPE register, it is permitted for implementations to have different numbers of MPU regions (including zero regions) in each security state.

 

MPU_CTRL.HFNMIENA

The behavior of the banked MPU_CTRL.HFNMIENA bit is modified such that it disables the MPU if the requested priority for the security domain the MPU is associated with is negative. This ensures the expected behavior for each security domain independent of the configuration of the other domain (E.G. MPU_CTRL.HFNMIENA can disable the non-secure MPU if FAULTMASK_NS is set regardless of whether AIRCR.PRIS is restricting the actual execution priority ot 0x80).

 

Security attribution

The security extension define 3 levels of memory security attribution, in ascending order of security they are:

1) Non-secure

2) Secure and non-secure callable

3) Secure and not non-secure callable

 

All boundaries between address ranges with different security attributes must be aligned to 32 byte boundaries. The attribution information can be provided by two diferent units:

l A security attribution unit (SAU) inside the processor.

l An IMPLEMENTATION DEFIENED attribution unit (IDAU) external to the processor. Whether this unit is present is IMPLEMENTATION DEFINED.

 

The attribution information from the SAU is used unless the IDAU specifies attributes with a higher security, in which case the IDAU attributes override the SAU attributes. The following address ranges are exempt from memory attribution by both the SAU and IDAU and have fixed behavior:

l 0xF0000000 -> 0xFFFFFFFF: If the processor implements the security extensions this memory range is always marked as secure and not non-secure callable for instruction fetches. If the security extensions are not present this range is marked as non-secure.

l The address ranges listed below are marked with the security state indicated by NS-Req (I.E. the current state of the processor for non-debug accesses). This effectively makes them exempt from security violation checking, and sets NS-Attr to NS-Req:

n  oxE0000000 -> 0xE0002FFF: ITM, DWT, FPB

n  0xE000E000 -> 0xE000EFFF: SCS range

n  0xE002E000 -> 0xE002EFFF: SCS NS alias range

n  0xE0040000 -> 0xE0041FFF: TPIU, ETM

n  0xE00FF000 -> 0xE00FFFFF: ROM table

n  0xE0000000 -> 0xEFFFFFFF: for instruction fetch only

n  Additional address ranges specified by the IDAU

 

Syndrome information associated with the SecureFault is reported in SFSR and SFAR.

 

 

Security attribution unit

The SAU configuration defines a number of memory regions, the number of regions is IMPLEMENTATION DEFINED and is discoverable by reading the SAU_TYPE.SREGION field. These regions are named SAU_REGIONn, where n is a number from 0 to SAU_TYPE.SREGION-1. The configuration fields associated with these regions can’t be directly accessed, but are indirectly accesses via window registers as shown in the following table:

SAU region configuration field

Associated window register field

SAU_REGIONn.ENABLE

SAU_RLAR.ENABLE

SAU_REGIONn.NSC

SAU_RLAR.NSC

SAU_REGIONn.BADDR

SAU_RBAR.BADDR

SAU_REGIONn.LADDR

SAU_RLAR.LADDR

When a memory access is performed the SAU is queried. By default memory is marked as being secure, unless the address matches a region with SAU_REGIONn.ENABLE set to 1 and SAU_REGIONn.NSC set to 0, in which case the memory is marked as being non-secure. An address is defined as matching a region in the SAU if SAU_REGIONn.BADDR <= Address <= SAU_REGIONn.LADDR. Any address that matches multiple SAU regions is marked as secure, and not non-secure callable (I.E. NSC=0) regardless of the attributes specified by the regions that matched the address.

 

Clearing the SAU_CTRL.ENALBE field results in the following SAU behavior:

l Addresses are not checked against the SAU regions

l The attribution of the address space is determined by the SAU_CTRL.ALLNS field

 

The configuration registers associated with the SAU are only accessible from a secure privileged state, accesses from unprivileged states generate a BusFault and access from the privileged non-secure state behave as RAZ/WI.

To permit lockdown of the SAU configuration it is IMPLEMENTATION DEFINED whether the SAU registers are writable.

 

The following specs of SAU_CTRL, SAU_TYPE, SAU_RNR, SAU_RBAR, SAU_RLAR

SAU_CTRL

Security attribution unit control register, address 0Xe000edd0, Resets to 0x0 or IMPLEMENTATION DEFINED

ENABLE (bit[0]) When set to 1 the SAU is enabled

ALLNS (bit[1]) When SAU_CTRL.ENABLE is 0 this bit controls if the memory is marked an non-secure or secure, and is enumerated as follows:

      0=Memory is marked as secure and is not non-secure callable

      1=Memory is marked as non-secure.

SAU_TYPE

Security attribution unit type register, address 0xE000EDD4, Read only

SREGION (bits[7:0]) The number of implemented SAU regions. If this field reads-as-zero the processor SAU behaves as follows:

n  SAU_CTRL.ENABLE behave as RAZ/WI

n  It is IMPLEMENTATION DEFINED whether SAU_CTRL.ALLNS behaves as RAO/WI (ie all attribution is performed by the IDAU)

n  SAU_RNR, SAU_RBAR, SAU_RLAR behave as RAZ/WI

SAU_RNR

Security attribution unit region number register, address 0xE000EDD8, Resets to UNKNOWN

REGION (bits[7:0]) Selects the region currently accessed by the SAU_RBAR and SAU_RLAR registers.

SAU_RBAR

Security attribution unit region base address register.

SAU_RLAR

Security attribution unit region limit address register.

ENABLE (bit[0]) When set to 1 this bit indicates the region is enabled

NSC (bit[1]) Non-secure callable. When NSC is set to 1 the region is marked as callable from the non-secure state. Any branches that cause a transition to the secure state that don’t target an instruction that is in a region with NSC set to 1 will result in a fault being raised. NOTE: the fault will be raised if either the upper or lower half of the instruction is not in an NSC region. NOTE: Normally a region defines an aera of the address space as non-secure, however regions with NSC set to 1 are secure.

This provides an additional level of protection to the SG guard instruction and can be used to eliminate the risk associated with the guard instruction op code being present in literal pools. To reduce the number of SAU regions required it would be advantageous to group all the entry points to the secure code together in a single region. Code in this region could be compiled as execute only and thus would not contain any literal pools

LADDR (bits[31:5]) Bits 31:5 of the limit address of the region, NOTE: bits[4:0] of the limit address are defined as 0x1F.

Implementation defined attribution unit

The IDAU can provide the following security attribution information:

l Security attribution exempt – Specifies if the address queried is exempt for security attribution. This information is combined with the address ranges that the architecture mandates are exempt from attribution.

l Non-secure – Specifies if the address queried is secure or non-secure.

l Non-secure callable – Specifies if code at the address queried can be called from the non-secure state. This attribute is only valid if the address is marked as secure.

l Region number – The region number the address matches. This value is only used by the TT instruction.

l Region number valid – Specifies that the region number is valid, this field has no effect on the attribution of the address and is only used by the TT instruction.

The non-secure and non-secure callable attributes are combined with the results from the SAU, with the resulting attribution being the least permissive of the two.
